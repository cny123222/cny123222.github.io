计算机网络知识点整理(2)：应用层

Last updated on March 24, 2025 am

这是SJTU-CS3611《计算机网络》课程的知识点整理系列。本文整理部分为“第2章：应用层”。

2.1 应用层协议基本原理

2.1.1 网络应用体系结构

  • ​客户-服务器体系结构 （如HTTP/FTP/IMAP）
    • 服务器：永久在线、固定IP、集中式管理
    • 客户端：间歇性连接、动态IP、不直接通信
  • ​P2P体系结构（如BitTorrent）
    • 无中心服务器、节点兼具客户端与服务端角色
    • 优势：自扩展性强、成本低
    • 挑战：动态IP、管理复杂

2.1.2 进程通信

  • ​套接字（Socket）​：应用层与传输层接口
  • 进程寻址：IP地址+端口号（HTTP:80/SMTP:25）

2.1.3 运输层服务选择

特性 TCP UDP
可靠性 可靠（重传机制） 不可靠
连接方式 面向连接（三次握手） 无连接
流量控制 有（滑动窗口） 无
拥塞控制 有（慢启动/拥塞避免） 无
适用场景 Web/Email/文件传输/远程终端访问 实时音视频/P2P文件分享
  • TCP 和 UDP 都不提供定时、吞吐量、安全性。

2.2 HTTP 协议（TCP / C-S架构）

2.2.1 核心机制

  • ​无状态性
    • 服务器不保存客户端状态，通过Cookie维持会话
    • ​Cookie工作流程：Set-Cookie 头 → 客户端存储 → 后续请求携带Cookie
  • ​连接模式
    • 非持续连接（HTTP/1.0）：每个对象需单独连接
    • 持续连接（HTTP/1.1+）：减少TCP握手开销（2RTT → 1RTT优化）
      • 分为 非流水线 和 流水线 两种

2.2.2 报文结构

  • ​请求报文

    • 方法：GET（获取资源）/ POST（提交表单）/ HEAD（获取首部）

  • ​响应报文

    • 状态码：200（成功）/ 301（重定向）/ 400（错误请求）/ 404（未找到）

2.2.3 缓存机制

  • ​Web缓存作用

    • ​降低延迟：就近获取资源减少传输时间
    • ​减少带宽消耗：避免重复下载相同内容
    • ​缓解服务器压力：减少直接请求源服务器的次数

  • 条件GET验证

  • 服务器返回304时，客户端使用本地缓存副本

  • ​缓存命中率计算

    • 公式：总延迟 = 命中率×缓存延迟 + (1-命中率)×源服务器延迟

2.2.4 HTTP演进

  • HTTP/2
    • 帧调度机制（解决HOL阻塞）、服务器推送、优先级
  • ​HTTP/3
    • 基于QUIC（UDP实现可靠传输）

2.3 电子邮件（TCP / C-S架构）

2.3.1 核心协议

  • ​SMTP（发件协议）​
    • 命令交互式（HELO/MAIL FROM/RCPT TO/DATA）
    • 7位ASCII限制
  • ​POP3/IMAP（收件协议）​
    • POP3：下载即删除，无状态
    • IMAP：服务器存储状态，支持文件夹管理

2.3.2 邮件报文格式

  • MIME扩展：支持多媒体附件

2.4 DNS（UDP 为主 / C-S架构）

2.4.1 分层结构

  • 根DNS →\rightarrow→ 顶级域(TLD) DNS（.com/.org） →\rightarrow→ 权威DNS →\rightarrow→ 本地DNS

2.4.2 查询过程

  • ​递归查询：本地DNS全权代理
  • ​迭代查询：每一步返回下一跳地址

2.4.3 资源记录类型

  • A记录（域名 → IPv4地址）
  • NS记录（域名 → 权威DNS服务器）
  • CNAME记录​（别名 → 规范名）
  • MX记录（域名 → 邮件服务器）

2.4.4 DNS安全

  • ​DDoS攻击：向根域名服务器发送大量流量
  • ​欺骗攻击：拦截DNS查询，返回虚假响应

2.5 P2P文件分发（UDP为主 / P2P架构）

2.5.1 BitTorrent核心机制

核心组件

  • ​Tracker服务器
    • 维护活跃节点列表
    • 响应节点查询请求

分块策略

策略 原理 优势
​最稀缺优先 优先下载全网最少副本的分块 加速文件完整传播
​随机块选择 随机选择分块下载 快速积累初始分块，降低冷启动延迟

节点协作

  • ​Tit-for-Tat
    • 优先服务上传速度最快的4个邻居
    • 每10秒重新评估邻居节点
  • ​乐观解封
    • 每30秒随机选择一个节点提供带宽
    • 发现潜在高速节点，鼓励新节点加入

2.5.2 分发效率公式推导

C/S模型

TCS=max⁡(NFus,Fdmin)T_{CS} = \max\left( \frac{NF}{u_s}, \frac{F}{d_{min}} \right) TCS​=max(us​NF​,dmin​F​)

  • NNN：客户端数量
  • FFF：文件总大小
  • usu_sus​：服务器上传带宽
  • dmind_{min}dmin​：最慢客户端下载带宽

P2P模型

TP2P=max⁡(Fus,Fdmin,NFus+∑i=1Nui)T_{P2P} = \max\left( \frac{F}{u_s}, \frac{F}{d_{min}}, \frac{NF}{u_s + \sum_{i=1}^N u_i} \right) TP2P​=max(us​F​,dmin​F​,us​+∑i=1N​ui​NF​)

  • uiu_iui​：第i个客户端的平均上传带宽

效率对比结论

模型 扩展性 服务器压力 适用场景
C/S 差（随N线性增长） 极高 小规模、私密文件
P2P 优（自扩展） 极低 大规模公开分发

2.6 视频流与内容分发网（HTTP / TCP / C-S架构）

2.6.1 DASH技术（Dynamic Adaptive Streaming over HTTP）

技术原理

  • ​视频分片：将视频切割为多个2~10秒的片段（不同码率版本并存）
  • ​Manifest文件：描述分片的URL、码率、分辨率等元数据

客户端决策流程

  1. ​初始带宽探测：首次请求低码率分片以测试网络状况。
  2. ​动态码率调整：
    • 带宽充足时，切换更高码率分片。
    • 带宽不足时，降级至低码率分片。
  3. ​缓冲区管理：维持5~10秒播放缓冲防止卡顿。

2.6.2 CDN（内容分发网络）

核心策略

策略 部署方式 优点 缺点 典型厂商
Enter Deep 边缘节点深入ISP网络 低延迟、高可用性 部署成本高 Akamai
Bring Home 集中部署在骨干网关键节点 维护成本低 跨ISP延迟较高 Limelight

缓存机制

  • ​主动推送
    • 预加载热点内容到边缘节点
    • 使用流行度预测算法（LRU/LFU）
  • ​动态路由
    • 用户DNS查询重定向至CDN的DNS服务器
    • 基于用户IP、节点负载、网络状态返回最优节点IP

2.7 套接字编程

2.7.1 UDP套接字

  • 无连接、不可靠、适用于实时应用

2.7.2 TCP套接字

  • 面向连接、可靠传输

2.7.3 ​API核心操作

  • 服务端：bind() → listen() → accept()
  • 客户端：connect() → send() / recv()
